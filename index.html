Why?

Why should we bother writing (automated) tests?

To Verify

A must to ensure we are providing value, yet;

  • This often happens haphazardly and is performed inconsistently.
  • It is onerous to record the how or what of manual verifications.
  • It is difficult to verify whether any shortcuts were taken.

Without verification we lack confidence that we delivered the customer's needs. When done manually we risk not having the proper verification taking place.

With automated tests, you can navigate these murky waters:

  • With programmatically defined tests, a single run will consistently perform all tests and can even automatically trigger them happening on code changes/pushes.
  • The programmatic tests will document the steps & verifications performed and also provide a report of any that failed.
  • The programmatic test runner will consistently run tests the same way every time.

To Protect

Tests can protect us from value loss by validating:

  • Previously delivered functionality still functions.
  • Previously fixed bugs have not been reintroduced.

When software of value has been created, there is always the potential for that value to be lost.

You can have automated tests triggered on code pushes, or run them manually at any time to have quick feedback before value loss makes it to a customer.

To Document

Documentation is vital, but often does not track with new changes. When tests are written well, they can serve as living documentation.

For example:

  • Functional tests document use cases.
  • Unit tests document design.

When your tests serve as documentation, they are guaranteed to never go out of date!

They can show others how to use your software's API, and when written well, can read like a narrative showing the dependencies, use cases and giving examples of how to engage with your software.

To Focus

Focus work on what is needed to meet the current needs instead of future possibilities.
